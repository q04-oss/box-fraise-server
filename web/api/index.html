API

Box Fraise runs a minimal login primitive at /v1/oauth. It exists so third-party services can accept "Sign in with Box Fraise" — issuing an ES256 JWT signed by our key, verifiable offline via our public JWKS.

Experimental · Unsupported

The signing key is regenerated on every deploy. Any JWT you cache will fail to verify after our next restart. There is no client registration, no consent flow, no discovery document, no refresh tokens.

Do not integrate this into a real product. When there is a real partner waiting for it, this hardens into a full OpenID Connect surface — persistent keys, key rotation, discovery, consent, all of it. Until then it exists to prove the interface shape.

Endpoints

POST/v1/oauth/token
Exchange a Box Fraise session (Bearer) for an ES256 JWT keyed to a caller-specified audience. Claims: iss, sub, aud, iat, exp, tier, verified. 15-minute lifetime.
GET/v1/oauth/jwks
Public. Returns the current JSON Web Key Set. One entry, ES256 / P-256. kid matches what tokens are signed with. Regenerated every deploy.
GET/v1/oauth/userinfo
Auth: a JWT from /oauth/token. Returns the same claims (sub, tier, verified) with live values from the current server state, rather than the values baked into the token when it was issued.

Example

# Exchange your Box Fraise session for a partner-audience JWT.
curl -X POST https://fraise.box/v1/oauth/token \
  -H "Authorization: Bearer $BF_SESSION" \
  -H "Content-Type: application/json" \
  -d '{ "audience": "https://your-service.example" }'

# The response includes access_token, expires_in, and a warning field.

# Verify the JWT with the public key from JWKS.
curl https://fraise.box/v1/oauth/jwks

Why unsupported

The Box Fraise project is small and pre-userbase. Maintaining a production login API requires stability guarantees, security patches, and integrator responsiveness the project can't honor yet. Shipping something unmaintained as if it were production would erode trust more than not shipping it. So this exists, publicly documented, honestly labelled.

The plan for when this hardens lives on the vision page. If you're a partner with a specific use case, this line will move — reach out.
